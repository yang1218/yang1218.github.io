Git基础开发流程

  • 拉代码（只需要在本地没有项目时，按顺序执行一次即可）
    1. git clone git@地址 克隆代码
    2. cd 目录 进到项目目录
    3. git remote 查看项目的远程分支
    4. git remote add 分支 git@地址 添加主项目的远程分支
    5. git remote 查看项目的远程分支
  • 日常操作（每次提交修改都应按照如下步骤操作）
    1. git stauts 查看本地文件修改情况
    2. git diff 查看文件的修改细节
    3. git add 文件名/git checkout 文件名 add 将文件添加至git暂存，checkout 回退文件的修改
  1. git commit -m “修改描述” 将代码提交至本地
  2. git pull 主项目远程分支 本地分支 将远程分支的改变同步到本地。如果有冲突，合并冲突后回到第一步重新提交

  3. git push 将本地的修改推送到自己远程项目

    说明：

    开发时尽量使用开发分支；
    步骤如下：git checkout -b imgCheck 创建名为imgCheck的分支，并切换到imgCheck
    上述的1-4步
    git checkout master 切换到master分支
    git merge imgCheck 将imgCheck的内容并到master分支上
    上述5-6步
    当功能开发完了可用 git branch -d imgCheck删除本地分支

    • 其他

  4. 同步远程分支操作

    1. git remote show origin 查看远程分支
    2. git remote update 更新本地的remote
    3. git fetch 相当于从远程获取最新版本到本地，不会自动marge
    4. git checkout -b dev –track upstream/dev 拉远程的dev分支并创建本地分支dev
    5. git push origin dev:dev 提交本地dev分支，并在自己的远程仓库不存在dev时创建dev分支

  5. git回滚到某个特定提交的操作

    1. git log 查看要回滚版本的哈希值
    2. git reset 版本号的哈希值（revert）

    3. git revert(前进)和git reset(回退)的区别

      • git revert是用一次新的commit来回滚之前的commit，git reset是直接删除指定的commit。
      • 在回滚这一操作上看，效果差不多。但是在日后继续merge以前的老版本时有区别。因为git revert是用一次逆向的commit“中和”之前的提交，因此日后合并老的branch时，导致这部分改变不会再次出现，但是git reset是直接把某个commit在该branch上删除，因而和老的branch再次merge时，这些被回滚的commit应该还会被引入。
      • git reset是把HEAD向后移动了一下，而git revert是HEAD继续前进，只是新的commit的内容和要revert的内容正好相反，能够抵消要被revert的内容。
